## TL;DR

Migration of the entire MonzaHaus product surface from the v2.0 Burgundy palette to the v2.1 Heritage Lavender palette per the official brand manual. Covers: CSS tokens, Saira wordmark, in-page Header, favicon, OG images, PDF reports, Excel exports, on-page report renderer, model/variant/admin pages, and the social-engine carousel templates.

After this merge, every visible MonzaHaus surface — the live site (light + dark), share previews, downloadable PDFs/Excel, and the carousel post generator — uses the same lavender system.

**Recommended merge:** `Create a merge commit` or `Rebase and merge`. **Do NOT squash.** The 9 commits are deliberately granular so a future bisect can pinpoint a regression to one specific surface (tokens, wordmark, exports, etc).

**Order of operations with the other open branches:** This branch is independent of `remove-onboarding-friction` and `otros-ajustes-front-end`. Either merge order works; there's no shared file beyond the layout and Header where conflicts are trivial line-level swaps.

## Naming pragmatism — why some token names look stale

Token aliases like `--color-salon-rose`, `--color-monza-pink`, and `--color-obsidian` still exist in `globals.css` and `social-engine/brand-tokens.css`. **Their values are now v2.1 lavender / noir** — only the names are stale. The 27 existing consumer call sites resolve them correctly without any further edits, so we kept the names to avoid a noisy 600-line rename in the same commit as the visual rebrand. A purely-cosmetic rename pass (rose → lavender, monza-pink → heritage-lavender, obsidian → noir) is a low-risk follow-up if anyone wants the semantic alignment. It's NOT blocking anything visual.

New code should prefer the v2.1 semantic tokens added in commit 1: `--lavender`, `--lavender-deep`, `--lavender-mid`, `--lavender-veil`, `--lavender-ink`, `--lavender-ink-deep`, `--ink`, `--noir`, `--noir-card`, `--cream`, `--bone`, `--stone`.

## Backend / designer follow-ups (not blocking)

These items showed up during the rebrand but don't gate the merge:

1. **PNG favicons** (`favicon-16/32/192/512.png`, `apple-touch-icon.png`) — regenerate from the new helmet SVG to preserve quality. Until then, modern browsers see the SVG correctly; only iOS home-screen icons and old Android Chrome show the legacy "M" momentarily.
2. **Token name cleanup** — `--color-salon-rose`, `--color-monza-pink`, `--color-obsidian` etc. carry v2.1 values but legacy names. Pure-cosmetic rename to `--lavender-*` / `--noir`. Low risk, can be done in any future PR.
3. **Saira in OG images** — `next/og` uses system-ui because shipping a font file into the edge runtime is heavier than the visual win. The wordmark in OG images uses bold sans-serif at the same letter-spacing, so the look is close. If you want pixel-perfect Saira in shareable previews, we can ship the Saira regular weight as a data URI and load it via `ImageResponse({fonts: [...]})`.

## If something breaks

| Symptom | Likely cause | Mitigation |
|---|---|---|
| Saira doesn't load (wordmark falls back to system-ui) | next/font fetch failed at build (rare); check Vercel build logs for `next/font` warnings | Re-run build. Saira is a Google Font, very stable. |
| Favicon still shows the "M" | Browser cache; the `<link rel="icon">` chain prefers SVG but cached PNGs win for ~24 hours | Hard refresh (`Cmd+Shift+R`); verify in private window. |
| A surface still looks burgundy | Token alias not migrated (regression) | `git grep` for `#7A2E4A`, `#D4738A`, `#0E0A0C`, `#161113` — should return zero hits across `src/`. If one resurfaces, file the path and we patch it. |
| Hydration warning re-appears in console | A new component nested `<a>` inside the BrowseCard's `View on` anchor | Inspect the warning's component path; the offender is whichever child added an `<a>`. |
