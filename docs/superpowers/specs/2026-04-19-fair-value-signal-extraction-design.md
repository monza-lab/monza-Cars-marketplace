# Fair Value + Signal Extraction — The Paid Investment Report

**Date:** 2026-04-19
**Status:** Draft — pending user review
**Supersedes:** Extends `2026-03-17-investment-report-real-data-design.md` (does not replace; builds on top of its market-stats foundation)
**Owner:** Edgar / Monza Haus

---

## 1. Problem

The paid **Haus Report** (product-facing name) is the core revenue and credibility unit of Monza Haus. Today it has two structural weaknesses:

1. **Fair Value is market-wide, not car-specific.** The current `fair_value_low/high` = P25–P75 of sold comparables for the series. It ignores everything that makes *this particular VIN* worth more or less: options, history, ownership, service records, originality. A 992 GT3 with Paint-to-Sample and 14 service stamps gets the same range as a neglected one.

2. **`investment_grade` (AAA/AA/A/B+/B/C)** is subjective, generated by Gemini without traceable methodology, and borrowed from credit-rating language that doesn't match the Porsche buyer's vocabulary. It undermines the "100% objective" promise that is Monza Haus' credibility moat.

We also carry a secondary problem: **browsing the catalog can't afford to run LLM extraction on every listing** (tens of thousands of cars). The expensive work must be gated behind payment and cached.

## 2. Strategic Approach

Two-tier architecture, both free of the investment-grade concept, both built only on objective data:

| Tier | Audience | What runs | Cost per listing | Storage |
|---|---|---|---|---|
| **Free View** (browse, detail page) | All visitors | Pre-computed market stats only (P25–P75, medians, counts). Zero LLM, zero per-listing extraction. | ~$0 (cron-computed) | `marketStats` (existing) |
| **Haus Report** (on-demand) | Authenticated user paying 1 credit | Full pipeline: LLM extraction of description, deterministic parsers, modifier engine, evidence-linked signal list | Paid **once per VIN in the history of the platform** (cached in `listing_reports` + new `listing_signals`); all subsequent paid views of the same listing hit cache | `listing_reports` + `listing_signals` |

Unit economics: one user pays → report generated → every future paying user of that listing accesses the cached result. Compute bill stays bounded no matter how many users look at the same listing.

The AAA/AA/A grade is **removed from the product entirely** — free view and paid report alike. Replaced by factual indicators everywhere.

### 4.2 New section: "Signals Detected" (between Valuation and Market Context)

A structured list of every objective signal the pipeline extracted from this listing. Each row:

- **Signal name** (e.g., *Paint-to-Sample*, *Full service records*, *Single previous owner*)
- **Value detected** (e.g., *"Gulf Blue (PTS code Y5C)"*, *"14 stamps through 2025"*, *"1 prior owner, 5 years"*)
- **Modifier applied** (e.g., `+10%`, `+4%`, `+3%`)
- **Evidence link** — clicking opens a side panel showing the exact sentence in `description_text`, photo number, or structured field that produced the signal
- **Source type badge**: `Listing text`, `Structured field`, `Seller context`, `External data`

### 4.3 New section: "Data we couldn't verify" (end of report, before Verdict)

Signals the pipeline looked for and did not find, each phrased as a question for the seller:

> - **Service book**: not mentioned in listing → *"Ask the seller for documented service history"*
> - **Paint meter readings**: not shown in photos → *"Request paint meter measurements at PPI"*
> - **Accident disclosure**: no explicit statement → *"Ask seller to confirm no accident history in writing"*

This section is the honest counter-balance to "what we found." It turns absence-of-signal from a liability into a concrete next action for the buyer — which *is* the product.

### 4.4 Photos in the report

Standardized 10-photo gallery, prioritized in this order:
1. First 3 photos of the listing (seller's hero shots)
2. Odometer (if OCR detects)
3. Engine bay
4. Driver-side interior
5. VIN plate (if visible)
6. Service book / Kardex (if visible)
7. Remaining listing photos until 10 filled

If listing has < 10, show all + label `"Listing provides N photos"` (itself a data-quality signal). If > 10, discreet link `"View all N photos on [source]"` back to BaT/AS24/AutoTrader — we don't rehost.

## 5. Signal Extraction Pipeline (v1 = sources 1 + 2 + 5)

v1 scope: text + structured fields + seller context. Photos (CV/OCR) and Kardex/VIN decode are deferred to v1.1.

### 5.1 Source 1 — Structured listing fields (deterministic)

Already in Supabase `listings` table. Parse directly:

| Field | Signal produced |
|---|---|
| `mileage` | Mileage delta vs. comparable set median |
| `transmission` | Manual vs. PDK modifier |
| `year` | Year-within-generation modifier |
| `color_exterior` + `color_interior` | Checked against known PTS codes for series |
| `country` + `region` | Market spec (US / Euro / gray import) |
| `source` + seller name | Seller tier (specialist vs. generalist vs. private — see 5.3) |

### 5.2 Source 2 — Description text (LLM fact extraction with strict JSON schema)

Single call to Gemini with the full `description_text`. The LLM is a **fact extractor**, not a judge. JSON schema requires booleans/numbers/enums only — never prose ratings.

#### Schema v1 (extracted per listing):

```typescript
interface ExtractedSignals {
  // Factory options (boolean per option)
  options: {
    sport_chrono: boolean | null
    pccb: boolean | null
    burmester: boolean | null
    lwb_seats: boolean | null // lightweight bucket seats
    carbon_roof: boolean | null
    paint_to_sample: { present: boolean; color_name: string | null; pts_code: string | null } | null
    factory_rear_spoiler_delete: boolean | null
    // expandable per series
  }

  // Service history
  service: {
    records_mentioned: boolean
    stamps_count: number | null         // from text like "14 service stamps"
    last_major_service_year: number | null
    intervals_respected: boolean | null  // per OEM schedule
    dealer_serviced: boolean | null      // vs independent
  }

  // Ownership
  ownership: {
    previous_owners_count: number | null
    one_owner_claim: boolean
    years_current_owner: number | null
    collector_owned_claim: boolean       // "collector-owned", "enthusiast-owned"
    garage_kept_claim: boolean
  }

  // Originality
  originality: {
    matching_numbers_claim: boolean
    original_paint_claim: boolean
    repaint_disclosed: { repainted: boolean; panels_mentioned: string[] } | null
    accident_disclosure: "none_mentioned" | "no_accidents_claim" | "prior_accident_disclosed"
    modifications_disclosed: string[]    // list of modifications (aftermarket wheels, exhaust, etc.)
  }

  // Documentation
  documentation: {
    ppi_available: boolean               // "PPI available on request", "recent PPI"
    carfax_linked: boolean
    window_sticker_shown: boolean
    build_sheet_shown: boolean
  }

  // Warranty
  warranty: {
    remaining_factory_warranty: boolean | null
    cpo_status: boolean
  }

  // Meta
  listing_completeness: "high" | "medium" | "low"  // LLM judges descriptiveness of the text itself
  extraction_confidence: "high" | "medium" | "low" // LLM self-assessment
}
```

**Key prompt instructions** (see §10 for full prompt):
- "Extract only what is explicitly or near-explicitly stated in the text. If the text does not mention a field, return `null` or `false` — never guess."
- "Do not infer condition from adjectives like 'beautiful' or 'stunning'. Those are marketing language, not facts."
- "For numeric fields like `stamps_count`, extract only if a specific number appears in the text."

## 6. Modifier Library v1

Each modifier is a row in a **versioned code constant** (`src/lib/fairValue/modifiers.ts`), not database-stored. Upgrading a modifier = code change = code review = traceability.

| # | Modifier | Source | Initial % | Justification |
|---|---|---|---|---|
| 1 | Mileage delta vs. comparable set | Source 1 | Linear regression on comparable set | Pure data |
| 2 | Transmission (manual vs. PDK) | Source 1 | Δ of medians within comp set per series | Pure data |
| 3 | Year within generation | Source 1 | Regression on comp set | Pure data |
| 4 | Paint-to-Sample present | Source 2 | +8–12% (starts at +10%) | Hagerty notes + BaT PTS premiums; citation link shown |
| 5 | Service records complete | Source 2 | +3–5% (starts at +4%) | PCA service guidelines + specialist dealer consensus; citation |
| 6 | Single/low previous owners | Source 2 | +2–4% (≤2 owners), −2–3% (>3 owners) | Specialist dealer consensus; citation |
| 7 | Original paint claim + no repaint | Source 2 | +3–5% | Hagerty originality guidance; citation |
| 8 | Accident disclosure (negative) | Source 2 | −5–15% depending on severity | Market-standard disclosure discount |
| 9 | Modifications disclosed | Source 2 | −2 to −8% depending on reversibility | Collector preference for original; citation |
| 10 | PPI / Carfax / window sticker provided | Source 2 | +1–3% cumulative | Trust signal; listing-level |
| 11 | Warranty remaining / CPO | Source 2 | +2–4% | Factory protection value |
| 12 | Seller tier (specialist / premium) | Source 5 | +2–4% | Screening premium |

**12 modifiers total in v1.** Each one:
- Has a **named constant** in code
- Has a **citation** (link or reference) shown on tooltip hover in the UI
- Has a **version** (so we can track when a modifier's % changes)
- Is capped: no single modifier exceeds ±15% adjustment; total modifiers capped at ±35% (guardrail against runaway stacking)

### 6.1 The fair value formula

```
baseline        = median(comparable_set) in USD
adjustments     = sum of active modifier deltas (each a % of baseline)
specific_fv_mid = baseline × (1 + adjustments_total)
specific_fv_low = specific_fv_mid × 0.93
specific_fv_high = specific_fv_mid × 1.07

Market delta    = (listing_price - specific_fv_mid) / specific_fv_mid
```

The ±7% band around `specific_fv_mid` reflects residual condition uncertainty (since v1 doesn't inspect photos). In v1.1 when CV is added, this band tightens.

### 6.2 Adaptive comparable selection (per prior decision)

Already agreed in brainstorm: start strict, expand if < 5 comparables.

- **Strict (layer 1):** same series + same trim + year ±2 + mileage ±30% + same canonical market
- **Series-wide (layer 2):** same series + year ±3, any trim, cross-market
- **Family-wide (layer 3):** same family group (e.g., all 992 variants)

Report always labels which layer was used. Confidence tier (`high|medium|low`) derives from `layer_used` + `comparables_count`.

### 8.1 TypeScript interfaces (new file: `src/lib/fairValue/types.ts`)

```typescript
export interface SignalEvidence {
  source_type: "listing_text" | "structured_field" | "seller_context" | "external"

  key: string

export interface HausReport {
```

## 10. Gemini Prompt v2 + Testing Protocol

### 10.1 New prompt structure

Replace `buildVehicleAnalysisPrompt` with **two separate calls**:

1. **`buildSignalExtractionPrompt(listing)`** — pure fact extraction. Returns `ExtractedSignals` JSON (schema in §5.2). Temperature 0. No reasoning, no recommendations.
2. **`buildAnalysisPrompt(listing, extractedSignals, marketStats)`** — the investment analysis. Consumes the extracted signals + market stats as input context. Returns the existing `red_flags`, `key_strengths`, `critical_questions`, `yearly_maintenance`, etc. — everything *except* a letter grade. Temperature 0.3.

Splitting the calls isolates extraction (must be deterministic, machine-verifiable) from analysis (can be more creative within bounds).

### 10.2 Testing protocol (mandatory before wiring to UI)

Before declaring the Gemini integration done:

1. `GEMINI_API_KEY` present in `.env.local`. ✓ Confirmed 2026-04-19. Model `gemini-2.5-flash` validated working with JSON mode + temperature 0 + null-field discipline.
2. Select 3 test listings from `listings` table covering different profiles:
   - 992 GT3 with rich description (high-extraction case)
   - 991 Carrera with sparse description (low-extraction edge case)
   - 997 GT3 RS with potentially rare options (challenging case)
3. Run `buildSignalExtractionPrompt` against each. Validate:
   - Response parses as valid JSON
   - Schema fully matches `ExtractedSignals` type
   - All fields present (nulls OK for missing signals, but keys must be there)
   - No hallucinated values (every non-null field traceable to the description text)
4. Run `buildAnalysisPrompt` against each (using extraction output as input). Validate:
   - JSON parses
   - No letter grades appear anywhere in the response
   - Numeric fields (maintenance, insurance) are within plausible ranges for the car
5. Save the 3 outputs as fixtures in `src/lib/ai/__fixtures__/gemini-signals-*.json`. These become regression tests for future prompt changes.
6. If any of the above fails, iterate on the prompt — do not add downstream workarounds.

### 10.3 Error handling

If signal extraction fails (malformed JSON, API error after 1 retry):
- Save `listing_reports` row with market stats only + `signals_extracted_at = null`
- Report UI shows a banner: *"Signal extraction unavailable — showing market data only. [Retry]"*
- Credit is **not deducted** on extraction failure (per existing spec)

## 12. Out of Scope (deferred to v1.1+)

- **Photo-based signal extraction (CV / OCR):** odometer verification, service book parsing, VIN plate decode, paint meter readings, window sticker. All deferred but schema designed to accept them (`evidence_source_type = "photo"` already reserved).

- **User-provided condition override.** Explicitly rejected per feedback — we don't punt to user input.
- **Proprietary "Monza Haus Index".** Deferred until we have enough data to derive our own modifiers empirically (12–18 months of accumulated `listing_signals`).

## 14. Closed Decisions (resolved 2026-04-19)

All three open questions were resolved during the brainstorm. Recording here so the implementation plan can reference concrete decisions.

- **Modifier citation format:** **Public links only** for v1 (Hagerty article, PCA post, BaT sold listing, etc.). Internal Monza Haus notes are *not* acceptable as primary citations — if no public source is found for a modifier's %, the modifier is shelved until one exists. Rationale: auditability from outside the company is the credibility moat.
- **Product-facing name:** **"Haus Report"** (not "Investment Report"). Final copy for the teaser is in §3.5. No numeric signal count in copy. Internal TypeScript interface renamed from `InvestmentReport` to `HausReport`.
- **Legacy `listing_reports` rows with `investment_grade` set:** **Keep them.** Grade is hidden in UI only. Existing reports remain valid and usable for their market stats + risk flags + ownership economics. When the next paying user for a given listing triggers access, the pipeline checks `signals_extracted_at`: if null, run the new extraction pipeline and populate the new fields in-place (upsert, not insert-new). No mass backfill.
